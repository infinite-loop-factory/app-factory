# Schedule Scheduler-Port & Manifest Contract

> Generated by PM Agent | Session: oma-00mqfwvhs35unqyb52 | Date: 2026-06-16
> Source ADR: `.agents/results/architecture/adr-schedule-cross-vendor.md`

This is NOT a REST API — it is a **module-boundary contract**. It fixes the single
interface + data schema shared by the four OS adapters, the `schedule:run` wrapper,
and the `schedule:*` command group.

---

## 1. Manifest schema (SSOT) — global-only

Location: **always `~/.agents/schedule/schedules.json`** (no project/global split).
Because OS schedulers are user-global, the registry is unified to user-global too.
File mode `0600`, directory mode `0700`.

Self-contained tree:
```
~/.agents/schedule/
├── schedules.json          <- this schema (SSOT)
├── env/<id>                <- capturedEnv (0600, named keys only)
└── runs/<id>/<ISO-ts>.md   <- run logs
```

```jsonc
{
  "version": 1,
  "jobs": [
    {
      "id": "sch_<base32-12>",        // oma-assigned stable id (distinct from any OS cron job id)
      "cron": "0 9 * * *",            // standard 5-field, local time
      "agentId": "qa-reviewer",       // <agent-id> for agent:spawn
      "prompt": "review the diff",    // inline text
      "promptPath": null,             // or a file path (XOR with prompt)
      "vendor": "codex",              // agent:spawn -m <vendor> (null = auto-detect)
      "workspace": "/abs/path",       // agent:spawn -w (absolute cwd at registration) = project identity
      "projectLabel": "oh-my-agent",  // git root basename of workspace (else dir basename); for list display
      "recurring": true,              // false = fire once then self-remove
      "maxAgeDays": 30,               // recurring auto-expiry (0 = indefinite)
      "capturedEnvRef": "env/sch_x",  // relative path to the separated env file (null if none)
      "createdAt": "2026-06-16T..Z",
      "lastFiredAt": null,            // updated by the wrapper on each fire
      "osBackend": "launchd",         // synced backend (for drift detection)
      "osJobLabel": "dev.oma.sch_x"   // OS-side identifier (for remove/query)
    }
  ]
}
```

**Invariants:**
- `prompt` XOR `promptPath` (exactly one).
- No plaintext secrets in the manifest — secrets go only in the `0600` file referenced
  by `capturedEnvRef`, and only the explicitly named keys.
- `id` is owned by oma; stable across OS re-registration/migration. `osJobLabel` is backend-specific.
- `workspace`/`projectLabel` keep "which project registered/runs this" always traceable
  (central registry, per-project execution context).

## 2. SchedulerPort interface (OS adapters)

```ts
interface ScheduledJobSpec {
  id: string;            // manifest job.id
  cron: string;          // 5-field
  command: string[];     // logical argv to run on fire. Always ["oma","schedule:run","<id>"].
                         // Because OS schedulers run with no shell / a minimal PATH, the
                         // adapter materializes command[0]="oma" to an absolute path
                         // (launchd: absolute ProgramArguments[0] + EnvironmentVariables PATH).
  label: string;         // osJobLabel
  workspace: string;     // cwd on fire
}

interface SchedulerPort {
  /** Register (or idempotently update by label) one job in the OS scheduler. */
  upsert(spec: ScheduledJobSpec): Promise<void>;
  /** Remove an OS job by label. No-op if absent. */
  remove(label: string): Promise<void>;
  /** Return the oma-managed labels currently registered in the OS (for drift comparison). */
  listLabels(): Promise<string[]>;
  /** Whether this adapter can run on the current host (launchd? systemd --user? schtasks?). */
  isAvailable(): Promise<boolean>;
}
```

**Implementation mapping:**
| Implementation | upsert | remove | listLabels |
|------|--------|--------|------------|
| `LaunchdAdapter` (macOS) | write plist + `launchctl bootstrap` | `launchctl bootout` + delete plist | glob `~/Library/LaunchAgents/dev.oma.*.plist` |
| `SystemdAdapter` (Linux) | write `.service`+`.timer` + `systemctl --user enable --now` | `disable --now` + delete units | `systemctl --user` timer glob |
| `CrontabAdapter` (fallback) | edit `crontab -l` + reinstall (between oma marker block) | drop the line in the marker block | parse the marker block |
| `SchtasksAdapter` (Windows) | `schtasks /Create /TN dev.oma.<id>` | `schtasks /Delete /TN` | `schtasks /Query /FO CSV` filter |

Adapter selection priority: native (launchd/systemd/schtasks) -> `CrontabAdapter` when unavailable.

## 3. `oma schedule:run <id>` wrapper contract

The single entry point the OS job invokes. Responsibilities:

1. Look up `<id>` in the manifest (missing -> non-zero exit + stderr, which triggers OS job self-cleanup).
2. If `capturedEnvRef` is set, load it and merge into the child env.
3. Run `oma agent:spawn <agentId> <prompt|@promptPath> <generatedSessionId> -m <vendor> -w <workspace>`.
4. Write the result to `~/.agents/schedule/runs/<id>/<ISO-ts>.md` (centralized, not the project dir).
5. Update manifest `lastFiredAt`. If `recurring=false`, self-remove the job (`port.remove` + manifest delete).
6. If spawn fails due to expired credentials, **loud-fail**: non-zero exit, stderr `re-auth required: <vendor>`,
   optional oma-voice notification. Never treat it as a silent success.

## 4. CLI surface (`schedule:` namespace)

| Command | Signature | Behavior |
|--------|----------|------|
| `schedule:add` | `<agent-id> <prompt> (--cron <expr> | --every <phrase>) [-m <vendor>] [-w <path>] [--once] [--max-age-days <n>] [--env <KEYS>]` | add to manifest + `port.upsert` |
| `schedule:list` | `[--json]` | manifest x OS drift, **grouped by PROJECT (`projectLabel`)** (synced / missing-in-os / orphan-in-os) |
| `schedule:remove` | `<id>` | `port.remove` + manifest delete (+ delete env file) |
| `schedule:run` | `<id>` | wrapper (§3) — not normally invoked by hand; the OS calls it |
| `schedule:sync` | `[--prune]` | re-sync manifest -> OS; `--prune` removes orphan OS jobs |

- `--cron` and `--every` are mutually exclusive (exactly one). `--every` accepts NL intervals
  (`5m`, `2h`, `1d`, `every 20m`, `every 5 minutes`) and rounds to the nearest cron-expressible step.
- `-m/--vendor` values are the oma agent:spawn set: `antigravity`, `claude`, `codex`, `cursor`,
  `opencode`, `qwen`, `grok`, `pi` (null = auto-detect). There is no `gemini` vendor — the Gemini
  backend is `antigravity` (the `agy` CLI).
- The registry is always `~/.agents/schedule/` (global-only). There is no `--global` flag — schedule
  is inherently user-global. `schedule:add` captures the current cwd as `workspace` and its git root
  basename as `projectLabel`.

## 5. Security contract (QA gate)

- manifest / env file mode `0600`, directory `0700`.
- `capturedEnvRef` stores **only the explicitly named keys** (`--env KEY1,KEY2`). Never dump the whole
  environment; prefer non-secret values.
- `schedule:add` is trusted-path only (interactive user). Do not expose the registration path to
  untrusted input (a scheduled prompt is arbitrary deferred execution).
- `schedule:run` only executes an `<id>` that actually exists in the manifest. No arbitrary argv injection.
- OS adapters use `execFileSync(file, args[])` (no shell) for all scheduler invocations.
