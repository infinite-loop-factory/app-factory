# UI 프롬프트 사용성 분석 및 개선 제안

이 문서는 작성된 UI 프롬프트를 사용성 관점에서 분석하고 개선점을 제안합니다.

## 🔍 주요 발견 사항

### ✅ 잘 설계된 부분
1. GPS 기반 자동 출발지 선택 - 사용자 편의성 향상
2. 원형 버튼 레이아웃 - 시각적 계층 구조 명확
3. 노선 공식 색상 사용 - 일관성 및 인식성
4. 모바일 우선 설계 - 타겟 플랫폼에 적합

---

## ⚠️ 사용성 문제점 및 개선 필요 사항

### 1. 홈 화면 (화면 1) - 심각도: 높음

#### 문제점
1. **정보 계층 구조 불명확**
   - GPS 기반 출발지가 카드 형태로만 표시되어 시각적 강조 부족
   - "다른 곳에서 출발할까요?" 버튼이 우측 상단에 있어 발견하기 어려울 수 있음

2. **GPS 상태 피드백 부족**
   - GPS 활성화/비활성화 상태가 명확하지 않음
   - GPS 권한 거부 시 대체 UI 전환이 즉시 명확하지 않음

3. **사용자 의도와 프롬프트 불일치**
   - 사용자 시나리오: "도착 정거장을 선택(입력)하는 화면"이 첫 화면
   - 프롬프트: 출발역과 도착역이 동등한 위치에 표시됨
   - **도착역 선택이 더 강조되어야 함**

4. **순서 자유도 미반영**
   - 사용자 요구: "출발지점을 먼저 설정하든, 도착지점을 먼저 설정하든 무방"
   - 프롬프트: 출발역이 상단에 고정되어 있어 순서 자유도가 시각적으로 드러나지 않음

#### 개선 제안
```
홈 화면 레이아웃 재구성:
- 도착 정거장 선택을 메인 영역으로 강조 (큰 카드, 상단 배치)
- GPS 기반 출발역은 컴팩트한 정보 바 형태로 표시
  - "📍 강남역에서 출발" (작은 배지/태그 형태)
  - "변경" 버튼 (작은 텍스트 버튼)
- 출발/도착 순서를 시각적으로 동등하게 표현
  - 두 선택 영역을 나란히 배치하거나
  - 탭 형태로 "출발" / "도착" 전환 가능하게
```

---

### 2. 출발 정거장 선택 (화면 2) - 심각도: 중간

#### 문제점
1. **원형 버튼 레이아웃의 실용성**
   - 원형 버튼이 3-4개 주변에 배치되면 터치 영역이 겹칠 수 있음
   - 모바일 화면에서 원형 배치가 작은 화면에 부적합할 수 있음
   - 거리 순으로 크기가 달라지면 일관성 부족

2. **GPS 비활성화 시 일관성 부족**
   - GPS 활성화 시: 원형 버튼 레이아웃
   - GPS 비활성화 시: 검색 리스트
   - 두 화면이 완전히 다른 패턴이라 사용자 혼란 가능

3. **거리 정보 표시 부족**
   - 사용자 요구: "도보로 접근 가능한 거리 한정"
   - 프롬프트: 거리만 표시, 도보 시간 미표시

4. **방향 정보 부재**
   - 각 역의 방향(동서남북) 정보가 없어 직관성 부족

#### 개선 제안
```
출발 정거장 선택 레이아웃 개선:

옵션 A: 원형 + 리스트 하이브리드
- 상단: 원형 버튼 레이아웃 (GPS 활성화 시)
- 하단: "더 많은 역 보기" 버튼 → 전체 역 리스트로 전환
- GPS 비활성화 시: 검색 리스트 (일관성 유지)

옵션 B: 카드 그리드 레이아웃
- 가장 가까운 역: 큰 카드 (상단 중앙)
- 주변 역들: 작은 카드 그리드 (2열 또는 3열)
- 각 카드에 거리 + 도보 시간 표시
- 방향 아이콘 추가 (↑↓←→)

추가 정보:
- 각 역 카드에 "200m (도보 3분)" 형태로 표시
- 방향 표시: "북쪽 200m" 또는 방향 아이콘
- "현재 위치 재확인" 버튼 추가
```

---

### 3. 도착 정거장 선택 (화면 3) - 심각도: 낮음

#### 문제점
1. **검색 우선순위 불명확**
   - 사용자 요구: "항상 입력/선택을 해야함"
   - 프롬프트: 검색과 필터가 동등하게 표시됨
   - 검색이 더 강조되어야 함

2. **최근 선택 역의 위치**
   - 상단에 최근 선택이 있으면 검색 입력과 겹칠 수 있음

3. **GPS 기반 정렬 옵션 부재**
   - 사용자가 원할 경우 거리순 정렬 옵션이 있으면 좋음

#### 개선 제안
```
도착 정거장 선택 개선:
- 검색 입력 필드를 가장 상단에 크게 배치
- 최근 선택은 검색 결과가 없을 때만 표시
- 필터는 접을 수 있는 섹션으로 변경
- 정렬 옵션 추가: "거리순", "이름순", "노선순"
```

---

### 4. 경로 결과 화면 (화면 4) - 심각도: 중간

#### 문제점
1. **경로 비교의 어려움**
   - 여러 경로가 리스트로만 표시되면 비교하기 어려움
   - 핵심 지표(시간, 비용)가 각 카드 내부에 있어 한눈에 비교 불가

2. **GPS 기반 최적 경로 표시 부족**
   - 사용자 요구: "현재 위치 기준 최적 경로" 명확히 표시
   - 프롬프트: GPS indicator만 있어 강조 부족

3. **정렬/필터 옵션 부재**
   - 사용자가 원하는 기준으로 정렬할 수 없음
   - 특정 조건(환승 없음 등) 필터링 불가

4. **비용 정보의 시각적 강조 부족**
   - 사용자 요구: 비용 계산이 필수
   - 프롬프트: 비용이 "medium" 크기로만 표시됨

#### 개선 제안
```
경로 결과 화면 개선:

비교 테이블 뷰 옵션:
- 리스트 뷰 / 비교 뷰 토글
- 비교 뷰: 시간, 비용, 환승 횟수를 테이블 형태로 나란히 표시

GPS 최적 경로 강조:
- "📍 현재 위치 기준 최적" 배지 (큰 배지, 상단 고정)
- 해당 경로 카드에 특별한 테두리/배경색
- 자동으로 상단에 배치

정렬/필터 바:
- 상단에 정렬 옵션: "시간순", "비용순", "환승순"
- 필터 아이콘: 환승 없음, 특정 노선 제외 등

비용 정보 강조:
- 비용을 시간과 동등한 크기로 표시
- "1,400원"을 큰 숫자로, "원"을 작게
- 색상으로 구분 (예: 시간=파랑, 비용=초록)
```

### 5. 경로 상세 화면 (화면 5) - 심각도: 낮음

#### 문제점
1. **시각적 경로 표현의 한계**
   - 텍스트 기반 timeline만으로는 경로 이해가 어려울 수 있음
   - 실제 노선도 형태의 시각화가 더 직관적

2. **실시간 정보 통합 부족**
   - 사용자 요구: 실시간 열차 위치, 도착 시간
   - 프롬프트: 정적 정보만 표시

3. **대체 경로 접근 어려움**
   - 현재 경로가 불만족스러울 때 대체 옵션으로 돌아가기 어려움

#### 개선 제안
```
경로 상세 화면 개선:
- 노선도 형태의 시각적 경로 표시 (가능하면)
- 실시간 정보 섹션:
  - "다음 열차 도착: 2분 후"
  - "현재 위치: 강남역 → 역삼역 구간"
- 상단에 "다른 경로 보기" 버튼 추가
- 각 구간에 실시간 혼잡도 표시 (채택된 기능)
```

## 🎯 전반적 UX 개선 제안

### 1. 피드백 및 상태 표시

#### 누락된 피드백
- **로딩 상태**: API 호출 중 명확한 로딩 인디케이터
- **에러 상태**: GPS 오류, 네트워크 오류 시 명확한 메시지
- **빈 상태**: 검색 결과 없음, 경로 없음 등
- **성공 상태**: 경로 찾기 완료 시 시각적 피드백

#### 개선 제안
```
각 화면에 상태 표시 추가:
- 로딩: 스켈레톤 UI 또는 스피너
- 에러: 명확한 에러 메시지 + 재시도 버튼
- 빈 상태: 친절한 안내 메시지 + 액션 제안
- 성공: 미묘한 애니메이션 또는 하이라이트
```

---

### 2. 접근성 (Accessibility)

#### 누락된 접근성 요소
- 스크린 리더 지원 언급 없음
- 키보드 네비게이션 고려 없음
- 색상 대비 확인 없음
- 터치 영역 최소 크기만 언급 (44px)

#### 개선 제안
```
접근성 개선:
- 모든 버튼에 aria-label 추가
- 색상 대비 WCAG AA 기준 준수
- 키보드 네비게이션 지원 (웹)
- 터치 영역 44px 이상 보장
- 포커스 상태 명확히 표시
```

---

### 3. 정보 계층 구조

#### 문제점
- 핵심 정보(시간, 비용)와 부가 정보의 구분이 불명확
- 사용자가 빠르게 결정할 수 있는 정보가 강조되지 않음

#### 개선 제안
```
정보 계층 구조 명확화:
- 1차 정보 (가장 크게): 시간, 비용
- 2차 정보 (중간): 환승 횟수, 거리
- 3차 정보 (작게): 상세 정보, 부가 기능
- 시각적 구분: 색상, 크기, 간격으로 명확히 구분
```

---

### 4. 오류 처리 및 엣지 케이스

#### 누락된 케이스
- GPS 권한 거부 후 복구 방법
- 네트워크 오류 시 재시도
- 동일한 역 선택 시 처리
- 경로 없음 시 대안 제시

#### 개선 제안
```
오류 처리 시나리오 추가:
- GPS 권한 거부: "위치 권한이 필요합니다" + 설정으로 이동 버튼
- 네트워크 오류: "인터넷 연결을 확인해주세요" + 재시도 버튼
- 동일 역 선택: "출발역과 도착역이 같습니다" + 즉시 피드백
- 경로 없음: "경로를 찾을 수 없습니다" + 대안 제시 (예: 다른 역으로 환승)
```

---

### 5. 사용자 시나리오와의 일치성

#### 주요 불일치
1. **첫 화면의 목적**: 사용자는 "도착 정거장 선택"이 메인이라고 했으나, 프롬프트는 출발/도착이 동등
2. **순서 자유도**: 시각적으로 출발/도착 순서가 자유롭다는 것이 드러나지 않음
3. **GPS 우선순위**: GPS 기반 출발지가 너무 강조되어 도착지 선택이 덜 중요해 보일 수 있음

#### 개선 제안
```
시나리오 반영:
- 홈 화면: 도착 정거장 선택을 메인으로, 출발역은 컴팩트하게
- 출발/도착 순서 자유도: 두 영역을 나란히 배치하거나 탭으로 전환
- GPS 출발지: 정보성으로만 표시, 선택 가능하지만 강제하지 않음
```

---

### 6. 모바일 UX 최적화

#### 개선 필요 사항
- 한 손 조작 고려 부족
- 스와이프 제스처 활용 부족
- 하단 액션 버튼 위치 최적화 필요

#### 개선 제안
```
모바일 UX 최적화:
- 주요 액션 버튼을 하단에 배치 (엄지 영역)
- 스와이프로 뒤로가기 지원
- 풀다운 새로고침 (GPS 위치 재확인)
- 하단 네비게이션 바 고려 (홈, 검색, 즐겨찾기 등)
```

## 🎨 디자인 개선 제안

### 시각적 계층 구조
```
제안된 정보 계층:
1. 핵심 액션 (가장 크게): 경로 찾기 버튼, 도착역 선택
2. 주요 정보 (크게): 시간, 비용
3. 보조 정보 (중간): 환승 횟수, 거리
4. 상세 정보 (작게): 구간별 정보, 부가 기능
```

### 색상 사용 전략
```
제안된 색상 계층:
- 주 색상: 노선 공식 색상 (경로 표시)
- 강조 색상: GPS 최적 경로 (예: 금색 또는 특별한 색상)
- 중립 색상: 배경, 텍스트 (회색 계열)
- 액션 색상: 버튼 (예: 파란색 또는 노선 색상)
```
