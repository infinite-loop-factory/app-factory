## 1단계: Google Play Console 계정 생성

**⏱️ 소요 시간**: 30분 ~ 7일 (본인 인증 시간 포함)

### 1.1 계정 등록

1. [Google Play Console](https://play.google.com/console) 접속
2. Google 계정으로 로그인 (Gmail 필요)
3. "개발자 계정 만들기" 클릭
4. **등록비 $25 결제** (일회성, 평생 유효)
   - 신용카드 또는 직불카드 결제
   - 환불 불가이므로 신중하게 진행

### 1.2 개발자 유형 선택

**개인 개발자** vs **조직 개발자** 중 선택:

| 구분 | 개인 개발자 | 조직 개발자 |
|------|------------|------------|
| 표시 이름 | 본인 이름 또는 상호 | 회사명 |
| 본인 인증 | 신분증 제출 | D-U-N-S 번호 + 사업자등록증 |
| 심사 기간 | 1~3일 | 3~7일 |
| 추천 대상 | 1인 개발자, 프리랜서 | 법인 사업자 |

> **💡 팁**: 대부분의 경우 **개인 개발자**로 시작하는 것을 권장합니다. 나중에 조직으로 전환 가능합니다.

### 1.3 본인 인증 절차

Google은 최근 악성 앱 방지를 위해 본인 인증을 강화했습니다.

**개인 개발자 인증 요구사항**:
- 신분증 사본 업로드 (주민등록증, 운전면허증, 여권 중 1개)
- 본인 사진 또는 셀카 (신분증과 함께)
- 주소 확인 서류 (선택사항)

**제출 후 대기**:
- 심사 기간: 1~3일 (영업일 기준)
- 심사 결과는 이메일로 통보
- 거절 시 사유 확인 후 재제출 가능

> ⚠️ **주의**: 본인 인증이 완료되어야 앱 출시가 가능합니다. 미리 진행하세요!

### 1.4 결제 프로필 설정

앱 수익(광고 수익 등)을 받기 위한 설정입니다.

1. Play Console 좌측 메뉴 → **설정** → **결제 프로필**
2. 은행 계좌 정보 입력
3. 세금 정보 입력 (한국 거주자는 W-8BEN 양식 작성)
4. 최소 지급액: $100 (누적 수익이 이 금액을 넘으면 자동 송금)

> **💡 팁**: AdMob 수익도 동일한 계좌로 받을 수 있습니다.

## 2단계: Google AdMob 설정

**⏱️ 소요 시간**: 20분

### 2.1 AdMob 계정 생성

1. [Google AdMob](https://admob.google.com) 접속
2. Play Console과 동일한 Google 계정으로 로그인
3. "시작하기" 클릭
4. 국가 및 통화 선택 (대한민국, KRW)
5. 약관 동의

### 2.2 앱 등록

1. AdMob 대시보드 → **앱** 탭 → **앱 추가**
2. "앱이 이미 게시되었나요?" → **아니요** 선택 (아직 출시 전이므로)
3. 플랫폼: **Android** 선택
4. 앱 이름: **이지토킹 - 질문 카드** 입력
5. 패키지명: **com.infiniteloop.easytalking** 입력
6. "앱 추가" 클릭

### 2.3 App ID 발급

앱 등록 완료 후 **App ID**가 자동 생성됩니다.

- 형식: `ca-app-pub-XXXXXXXXXXXXXXXX~YYYYYYYYYY`
- 예시: `ca-app-pub-1234567890123456~9876543210`

> **📝 메모**: 이 App ID를 메모장에 복사해두세요. 코드에 적용할 예정입니다.

### 2.4 Banner Ad Unit 생성

이지토킹 앱은 5개 화면에 배너 광고를 표시합니다.

1. AdMob 대시보드 → 방금 등록한 앱 선택 → **광고 단위** 탭
2. **광고 단위 추가** 클릭
3. 형식: **배너** 선택
4. 광고 단위 이름: `Banner - Main` (또는 원하는 이름)
5. "광고 단위 만들기" 클릭

완료 후 **Ad Unit ID**가 생성됩니다.

- 형식: `ca-app-pub-XXXXXXXXXXXXXXXX/ZZZZZZZZZZ`
- 예시: `ca-app-pub-1234567890123456/1234567890`

> **💡 팁**: 화면별로 별도 Ad Unit을 만들 수도 있지만, 동일한 배너 ID를 재사용해도 됩니다.

### 2.5 코드에 ID 적용

발급받은 **App ID**와 **Ad Unit ID**를 코드에 반영합니다.

#### 2.5.1 `.env.production` 파일 수정

프로젝트 루트의 `.env.production` 파일을 열고 다음 값을 교체하세요:

```bash
# AdMob Configuration (PRODUCTION)
EXPO_PUBLIC_ADMOB_APP_ID_ANDROID=ca-app-pub-XXXXXXXXXXXXXXXX~YYYYYYYYYY  # 👈 여기에 App ID 입력
EXPO_PUBLIC_ADMOB_BANNER_ID_ANDROID=ca-app-pub-XXXXXXXXXXXXXXXX/ZZZZZZZZZZ  # 👈 여기에 Banner Ad Unit ID 입력
EXPO_PUBLIC_ADMOB_APP_ID_IOS=ca-app-pub-XXXXXXXXXXXXXXXX~YYYYYYYYYY
EXPO_PUBLIC_ADMOB_BANNER_ID_IOS=ca-app-pub-XXXXXXXXXXXXXXXX/ZZZZZZZZZZ
```

각 필드 설명:
- `EXPO_PUBLIC_ADMOB_APP_ID_ANDROID`: AdMob에서 발급받은 Android App ID
- `EXPO_PUBLIC_ADMOB_BANNER_ID_ANDROID`: 배너 광고 Ad Unit ID
- iOS 필드는 나중에 iOS 출시 시 사용

#### 2.5.2 `app.config.ts`의 androidAppId 교체 (line 51)

`app.config.ts` 파일을 열고 51번 줄 근처에서 다음 부분을 수정:

```typescript
// Before (기존 테스트 ID)
config: {
  googleMobileAdsAppId: 'ca-app-pub-3940256099942544~3347511713', // 테스트 ID
}

// After (실제 발급받은 App ID)
config: {
  googleMobileAdsAppId: 'ca-app-pub-XXXXXXXXXXXXXXXX~YYYYYYYYYY', // 👈 실제 App ID로 교체
}
```

> ⚠️ **중요**: `.env.production` 파일은 Git에 커밋하지 마세요! (`.gitignore`에 이미 포함되어 있음)

## 3단계: 개인정보 처리방침 호스팅

**⏱️ 소요 시간**: 30분

Google Play는 광고를 표시하거나 사용자 데이터를 수집하는 앱에 **개인정보 처리방침 공개 URL**을 요구합니다.

### 3.1 초안 확인

프로젝트의 `docs/PRIVACY_POLICY.md` 파일에 초안이 이미 작성되어 있습니다.

**수정 필요 항목**:
1. **연락처 이메일**: `[개발자 이메일]` → 실제 이메일 주소로 교체
   - 예: `support@easytalking.app` 또는 개인 Gmail
2. **마지막 업데이트 날짜**: 실제 출시일로 수정

### 3.2 호스팅 방법 (권장: GitHub Pages)

**가장 간단하고 무료인 방법**입니다.

#### 방법 1: GitHub Pages 사용

1. 프로젝트가 GitHub에 호스팅되어 있다면:
   - 저장소 → **Settings** → **Pages**
   - Source: **Deploy from a branch**
   - Branch: **main** (또는 master), Folder: **/docs**
   - Save

2. 5분 후 다음 URL에서 접근 가능:
   ```
   https://{github-username}.github.io/{repository-name}/PRIVACY_POLICY.html
   ```

3. `PRIVACY_POLICY.md`를 HTML로 변환:
   - [Markdown to HTML 변환기](https://markdowntohtml.com) 사용
   - 변환된 HTML을 `docs/PRIVACY_POLICY.html`로 저장
   - Git commit & push

> **💡 팁**: 저장소가 private이면 Pages가 작동하지 않습니다. Public으로 변경하거나 다른 방법 사용하세요.

#### 방법 2: Notion 공개 페이지

1. Notion에서 새 페이지 생성
2. `PRIVACY_POLICY.md` 내용을 복사하여 붙여넣기
3. 우측 상단 **공유** → **웹에 게시**
4. 생성된 URL 복사 (예: `https://notion.so/...`)

#### 방법 3: Google Sites (가장 쉬움)

1. [Google Sites](https://sites.google.com) 접속
2. 새 사이트 만들기
3. 제목: "이지토킹 개인정보 처리방침"
4. `PRIVACY_POLICY.md` 내용을 복사하여 붙여넣기
5. 우측 상단 **게시** 클릭
6. 생성된 URL 복사 (예: `https://sites.google.com/view/easytalking-privacy`)

### 3.3 최종 URL 확보

어떤 방법을 사용하든 **공개적으로 접근 가능한 URL**을 확보하세요.

- ✅ 좋은 예: `https://example.com/privacy-policy`
- ❌ 나쁜 예: `file:///Users/...` (로컬 파일 경로)

이 URL은 Play Console 앱 등록 시 입력합니다.

## 4단계: 스토어 등록 자료 준비

**⏱️ 소요 시간**: 2~3시간

### 4.1 스크린샷 촬영

Google Play 요구사항:
- **최소 2장, 권장 4~8장**
- 해상도: **1080x1920 (세로)** 또는 **1920x1080 (가로)**
- 형식: PNG 또는 JPG
- 최대 파일 크기: 8MB

#### 추천 화면 구성 (총 5장)

1. **홈 화면** (카테고리 선택 화면)
   - 앱의 첫인상을 보여주는 메인 화면
2. **난이도 선택 화면**
   - 쉬움/보통/어려움 선택 UI
3. **질문 카드 스와이프 화면**
   - 실제 앱의 핵심 기능 시연
4. **질문 리스트 화면**
   - 저장된 질문 목록 보기
5. **즐겨찾기/검색 화면**
   - 부가 기능 강조

#### 촬영 방법

**방법 1: Android 에뮬레이터 (권장)**

1. Android Studio → AVD Manager 실행
2. 기기: **Pixel 6** (해상도 1080x2400, 적절히 잘림)
3. 앱 실행: `npm run android`
4. 화면 캡처: 에뮬레이터 우측 툴바 → 카메라 아이콘

**방법 2: 실제 기기**

1. 개발자 옵션 활성화
2. USB 디버깅 연결
3. 화면 캡처: 볼륨 다운 + 전원 버튼 동시 누르기
4. 해상도가 1080x1920이 아니면 [Photopea](https://www.photopea.com)에서 리사이즈

> **💡 팁**: 스크린샷에 상태 바(시간, 배터리)를 제거하려면 에뮬레이터 사용을 권장합니다.

### 4.2 고해상도 아이콘 (512x512)

1. 프로젝트의 `assets/icon.png` 파일 확인 (1024x1024)
2. [Photopea](https://www.photopea.com) 또는 Photoshop에서 512x512로 리사이즈
3. PNG 형식으로 저장
4. 투명 배경 제거 (Google Play는 불투명 아이콘 요구)

> ⚠️ **주의**: 아이콘은 **정사각형**이어야 하며, 둥근 모서리는 시스템이 자동 처리합니다.

### 4.3 피처 그래픽 (1024x500)

앱 상단에 표시되는 가로형 배너 이미지입니다.

**제작 방법**:
1. [Canva](https://www.canva.com) 접속 (무료)
2. 사용자 지정 크기: **1024 x 500 px**
3. 템플릿 검색: "App Banner" 또는 직접 디자인
4. 포함 요소:
   - 앱 이름: "이지토킹"
   - 부제: "질문 카드로 대화 시작하기"
   - 앱 아이콘 (선택사항)
   - 배경색: 앱 테마 색상 사용 권장
5. PNG 또는 JPG로 다운로드

> **💡 팁**: 텍스트가 너무 작으면 가독성이 떨어집니다. 큰 폰트 사용을 권장합니다.

### 4.4 앱 설명 작성

#### 짧은 설명 (최대 80자)

Play Store 검색 결과에 표시되는 한 줄 설명입니다.

**예시**:


#### 자세한 설명 (최대 4000자)

앱의 모든 기능과 가치를 상세히 설명합니다.

**예시 템플릿**:
```markdown

✨ 주요 기능

❤️ 즐겨찾기
마음에 드는 질문은 즐겨찾기에 저장하여 나중에 다시 사용할 수 있습니다.

🔍 검색 기능
키워드로 원하는 질문을 빠르게 찾아보세요.

🎨 직관적인 UI
카드 스와이프로 쉽고 재미있게 질문을 탐색할 수 있습니다.

🔒 개인정보 보호
회원가입 불필요! 모든 데이터는 내 기기에만 저장됩니다.

• 회식 자리에서 아이스브레이킹이 필요할 때
• 가족 모임에서 재미있는 시간을 보내고 싶을 때
```

### 4.5 카테고리 선택

Google Play 카테고리 중 하나를 선택해야 합니다.

**추천 카테고리**:
1. **라이프스타일** (가장 적합)
2. **소셜**
3. **엔터테인먼트**

> **💡 팁**: 카테고리는 나중에 변경 가능하므로 너무 고민하지 마세요.

## 5단계: EAS 프로덕션 빌드

**⏱️ 소요 시간**: 15~30분 (빌드 대기 시간 포함)

### 5.1 EAS CLI 설치 확인

```bash
# EAS CLI가 설치되어 있는지 확인
npm list -g eas-cli

# 없다면 설치
npm install -g eas-cli
```

### 5.2 프로덕션 빌드 실행

프로젝트 루트 (`apps/question-card/`)에서 실행:

```bash
cd apps/question-card
eas build -p android --profile production
```

### 5.3 빌드 과정 설명

1. **Expo 로그인 프롬프트**
   - Expo 계정으로 로그인 (없으면 회원가입)

2. **프로젝트 초기화** (첫 빌드 시)
   - EAS 프로젝트 ID 생성
   - `app.json`에 자동 추가됨

3. **Keystore 생성** (첫 빌드 시)
   - "Would you like us to handle your keystore?" → **Yes** 선택
   - EAS가 자동으로 keystore 생성 및 관리
   - 이 keystore는 앱의 "서명"이며, 향후 모든 업데이트에 동일하게 사용됨

4. **빌드 큐 대기**
   - EAS 클라우드에서 빌드 시작
   - 진행 상황: [Expo 대시보드](https://expo.dev)에서 실시간 확인 가능
   - 예상 시간: 10~20분

5. **빌드 완료 알림**
   - 터미널에 다운로드 링크 표시
   - 이메일로도 알림 발송

### 5.4 AAB 파일 다운로드

빌드 완료 후 **AAB(Android App Bundle)** 파일을 다운로드합니다.

```bash
# 터미널에 표시된 URL에서 다운로드
# 또는 Expo 대시보드에서 다운로드 버튼 클릭
```

- 파일 형식: `build-xxxxxxxx.aab`
- 파일 크기: 약 30~50MB

> **📝 메모**: AAB는 APK와 달리 직접 설치할 수 없습니다. Play Store 전용 포맷입니다.

### 5.5 Keystore 백업 (중요!)

**⚠️ 필수 단계**: Keystore를 잃어버리면 앱 업데이트가 불가능합니다!

```bash
# Keystore 정보 확인 및 백업
eas credentials --platform android

# 화면에 표시되는 메뉴에서:
# 1. Select "View credentials in your project"
# 2. Download keystore 선택
# 3. 다운로드된 keystore 파일을 안전한 곳에 백업 (USB, 클라우드 등)
```

백업 체크리스트:
- [ ] keystore 파일 백업 (예: `easytalking-keystore.jks`)
- [ ] keystore 비밀번호 기록 (EAS가 자동 생성한 비밀번호)
- [ ] key alias 기록
- [ ] 안전한 저장소에 보관 (Google Drive, 1Password 등)

> **💡 팁**: 비밀번호는 EAS 대시보드 → Credentials 섹션에서 확인 가능합니다.

## 7단계: AAB 업로드 및 테스트 트랙

**⏱️ 소요 시간**: 30분

### 7.1 출시 트랙 이해하기

Google Play는 4가지 출시 트랙을 제공합니다.

| 트랙 | 대상 | 사용 시점 | 심사 |
|------|------|----------|------|
| **내부 테스트** | 최대 100명 (이메일 초대) | 초기 테스트, 크래시 확인 | 없음 (즉시 배포) |
| **비공개 테스트** | 초대받은 사용자만 | 베타 테스트, 피드백 수집 | 간소화 (1~2일) |
| **공개 테스트** | 누구나 참여 가능 | 대규모 베타 테스트 | 간소화 (1~2일) |
| **프로덕션** | 전체 공개 | 정식 출시 | 전체 심사 (1~7일) |

> **💡 권장**: 처음 출시는 **내부 테스트** → **프로덕션** 순서로 진행하세요.

### 7.2 내부 테스트 트랙에 업로드 (권장)

#### 7.2.1 새 출시 만들기

1. 좌측 메뉴 → **출시** → **테스트** → **내부 테스트**
2. **새 출시 만들기** 클릭

#### 7.2.2 AAB 업로드

1. **App Bundle 업로드** 클릭
2. [5단계](#5단계-eas-프로덕션-빌드)에서 다운로드한 AAB 파일 선택
3. 업로드 완료 대기 (1~2분)
4. 업로드 후 자동으로 앱 정보 분석:
   - 버전 코드: `app.json`의 `versionCode` 값
   - 버전 이름: `app.json`의 `version` 값
   - 지원 기기: 자동 탐지

> ⚠️ **오류 발생 시**: "중복된 버전 코드" 오류가 나면 `app.json`의 `versionCode`를 증가시킨 후 다시 빌드

#### 7.2.3 출시 노트 작성

```
첫 출시 - 이지토킹 질문 카드 앱

- 즐겨찾기 및 검색 기능
```

#### 7.2.4 출시 검토 및 배포

1. **검토** 클릭
2. 경고 메시지 확인 (있으면 수정)
3. **내부 테스트 시작** 클릭

**결과**: 즉시 배포됨 (심사 없음)

### 7.3 테스터 추가

1. **내부 테스트** 탭 → **테스터** 섹션
2. **이메일 목록 만들기** 클릭
3. 목록 이름: "내부 팀"
4. 테스터 이메일 추가 (최대 100개)
   - 예: `john@example.com, jane@example.com`
5. **저장** 클릭

#### 테스터 초대 링크

1. **복사 가능한 링크** 버튼 클릭
2. 링크를 테스터에게 공유
3. 테스터는 링크로 접속 → 앱 다운로드

### 7.4 내부 테스트 확인

테스터가 앱을 다운로드하고 테스트합니다.

**확인 사항**:
- [ ] 앱 실행 정상
- [ ] 광고 표시 정상
- [ ] 모든 기능 작동
- [ ] 크래시 없음

문제가 있으면 수정 후 새 AAB 업로드 (버전 코드 증가 필요)

### 7.5 프로덕션으로 승격 준비

내부 테스트에서 문제가 없으면 **프로덕션** 트랙으로 진행합니다.

> **💡 팁**: 비공개 테스트를 건너뛰고 바로 프로덕션으로 갈 수 있습니다.

### 8.4 단계적 출시 설정 (선택사항)

**단계적 출시(Staged Rollout)**란?
- 전체 사용자에게 한 번에 배포하지 않고, 일부 사용자에게만 먼저 배포
- 크래시나 버그 발생 시 피해 최소화

**설정 방법**:
1. **출시 개요** 섹션에서 **단계적 출시 사용** 체크
2. 초기 비율 선택:
   - 1% (권장: 첫 출시)
   - 5%
   - 10%
   - 50%
3. 24시간 후 문제 없으면 비율 증가 가능

> **💡 팁**: 첫 출시는 1%로 시작하여 점진적으로 늘리는 것을 권장합니다.

3. **프로덕션 출시 시작** 클릭
4. 확인 팝업 → **출시** 클릭

### 8.6 Google 심사 대기

제출 후 Google의 자동 및 수동 심사가 진행됩니다.

**심사 단계**:
1. **제출됨**: 심사 대기열에 추가됨
2. **심사 중**: 자동 검사 (악성코드, 정책 위반 스캔)
3. **수동 심사**: Google 담당자가 앱 테스트 (1~7일)
4. **승인** 또는 **거절**

**심사 진행 상황 확인**:
- Play Console 대시보드 → **출시 개요**
- 이메일 알림

### 8.7 심사 거절 시 대응

거절 사유는 이메일로 통보됩니다.

**주요 거절 사유**:
| 사유 | 해결 방법 |

## 9단계: 출시 후 관리

**⏱️ 소요 시간**: 지속적

### 9.1 Play Console 대시보드 모니터링

정기적으로 확인해야 할 지표:

| 항목 | 위치 | 확인 주기 |
|------|------|----------|
| **설치 수** | 대시보드 → 통계 | 주 1회 |
| **평점/리뷰** | 대시보드 → 사용자 리뷰 | 주 2~3회 |
| **크래시율** | 대시보드 → Android Vitals | 매일 |
| **ANR율** | 대시보드 → Android Vitals | 주 1회 |
| **수익** | AdMob 대시보드 | 주 1회 |

### 9.2 크래시 보고서 확인

1. 좌측 메뉴 → **품질** → **Android Vitals** → **크래시 및 ANR**
2. 크래시 발생 시:
   - 스택 트레이스 확인
   - 재현 단계 파악
   - 코드 수정
   - 새 버전 출시

> **💡 팁**: 크래시율 2% 이상 시 Google이 경고를 보냅니다. 신속히 대응하세요.

### 9.3 사용자 리뷰 대응

**좋은 리뷰 대응 예시**:
```
이지토킹을 사랑해주셔서 감사합니다!
앞으로도 더 좋은 질문을 추가할 예정이니 기대해주세요 😊
```

**부정적 리뷰 대응 예시**:
```
소중한 피드백 감사합니다.
말씀하신 문제는 다음 업데이트에서 개선하겠습니다.
불편을 드려 죄송합니다.
```

> **💡 팁**: 모든 리뷰에 답변할 필요는 없지만, 부정적 리뷰는 반드시 응대하세요.

### 9.4 앱 업데이트 방법

새 기능 추가나 버그 수정 후 업데이트 절차:

#### 1. 버전 정보 업데이트

`app.json` 파일 수정:

```json
{
  "expo": {
    "version": "1.0.1",  // 버전 이름 증가
    "android": {
      "versionCode": 2   // 버전 코드 증가 (필수!)
    }
  }
}
```

**버전 규칙**:
- `versionCode`: 정수, 이전보다 **반드시 커야 함** (예: 1 → 2 → 3)
- `version`: 문자열, 사용자에게 표시 (예: 1.0.0 → 1.0.1 → 1.1.0)

#### 2. 새 AAB 빌드

```bash
eas build -p android --profile production
```

#### 3. Play Console에 업로드

1. **프로덕션** 트랙 → **새 출시 만들기**
2. 새 AAB 업로드
3. 출시 노트 작성 (변경 사항 설명)
4. **검토** → **프로덕션 출시 시작**

#### 4. 심사 대기

업데이트도 심사를 받지만, 일반적으로 첫 출시보다 빠름 (1~3일)

### 9.5 AdMob 수익 확인

1. [AdMob 대시보드](https://admob.google.com) 로그인
2. **보고서** 탭 → 일일 수익 확인
3. 예상 수익률:
   - 초기 (사용자 100명): 하루 $0.5~$2
   - 성장 후 (사용자 10,000명): 하루 $50~$200

> **💡 팁**: 광고 노출이 너무 많으면 사용자 경험이 나빠집니다. 적절한 균형 유지가 중요합니다.

### 9.6 정기적인 유지보수

| 작업 | 주기 | 내용 |
|------|------|------|
| **Expo SDK 업데이트** | 분기별 | `npx expo upgrade` |
| **의존성 업데이트** | 월 1회 | `npm outdated` 확인 |
| **보안 패치** | 즉시 | `npm audit` 경고 시 대응 |
| **새 질문 추가** | 월 1회 | 사용자 요청 반영 |

## 부록: 유용한 명령어 모음

### EAS 빌드 관련

```bash
# 프로덕션 빌드
eas build -p android --profile production

# 프리뷰 빌드 (개발용 APK)
eas build -p android --profile preview

# 빌드 상태 확인
eas build:list

# 특정 빌드 다운로드
eas build:download --id [BUILD_ID]
```

### EAS Credentials 관련

```bash
# Keystore 정보 확인
eas credentials --platform android

# Keystore 다운로드 (백업용)
eas credentials --platform android
# 메뉴에서 "Download credentials" 선택

# Keystore 수동 업로드 (복구 시)
eas credentials --platform android
# 메뉴에서 "Upload new credentials" 선택
```

### 로컬 테스트

```bash
# 개발 서버 실행
npm start

# Android 에뮬레이터에서 실행
npm run android

# iOS 시뮬레이터에서 실행 (Mac only)
npm run ios

# 프로덕션 환경 변수로 실행
ENV_FILE=.env.production npm start
```

### 프로젝트 정보 확인

```bash
# Expo 프로젝트 정보
npx expo config

# 의존성 확인
npm list

# 보안 취약점 확인
npm audit

# 오래된 패키지 확인
npm outdated
```

### Play Store 제출 자동화 (고급)

```bash
# EAS Submit으로 자동 업로드 (설정 필요)
eas submit -p android --latest

# 수동으로 AAB 지정
eas submit -p android --path ./build.aab
```

> **💡 팁**: `eas submit`을 사용하려면 Play Console API 설정이 필요합니다. 처음에는 수동 업로드를 권장합니다.

### 트러블슈팅 명령어

```bash
# 캐시 삭제 후 재빌드
npm start --clear

# node_modules 재설치
rm -rf node_modules && npm install

# Metro 번들러 캐시 삭제
npx expo start -c

# Android 빌드 캐시 삭제 (로컬)
cd android && ./gradlew clean && cd ..
```
